# Plan 3: Deploy, Nginx, SSL, and Go-Live

<objective>
Deploy the OVNI AI application to the EC2 instance, configure Nginx as a reverse proxy with SSL via Let's Encrypt (Certbot), set up PM2 for process management, and verify the full stack is running and accessible over HTTPS on the production domain.
</objective>

<must_haves>
- Application cloned, built, and running on EC2 via PM2
- Nginx configured as reverse proxy: HTTPS on port 443 to localhost:3000
- Valid SSL certificate (Let's Encrypt via Certbot) with auto-renewal
- Domain resolves to Elastic IP via Route53
- Health check returns 200 on https://<domain>/health
- Deploy script automates the full update cycle (pull, install, build, migrate, restart)
- OpenClaw Docker containers can reach metering proxy
</must_haves>

<task id="3.4">
<title>Configure Nginx reverse proxy</title>
<read_first>
- src/server.ts (listens on 0.0.0.0:3000)
- src/services/meteringProxy.ts (listens on :3001 — internal only)
- .planning/phases/01-aws-infrastructure-deploy/01-CONTEXT.md (decision D-11: Nginx terminates SSL, proxy to :3000)
</read_first>
<action>
Create the Nginx config file in the project for reference and deployment:

**File: `infra/nginx/ovni-ai.conf`** (committed to Git)
```nginx
# OVNI AI — Nginx Reverse Proxy Configuration
# Deployed to: /etc/nginx/sites-available/ovni-ai

# Rate limiting zone
limit_req_zone $binary_remote_addr zone=api:10m rate=30r/s;

# Upstream
upstream ovni_api {
    server 127.0.0.1:3000;
    keepalive 32;
}

# HTTP → HTTPS redirect
server {
    listen 80;
    server_name <DOMAIN>;

    # Let's Encrypt challenge
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

# HTTPS server
server {
    listen 443 ssl http2;
    server_name <DOMAIN>;

    # SSL certificates (managed by Certbot)
    ssl_certificate /etc/letsencrypt/live/<DOMAIN>/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/<DOMAIN>/privkey.pem;

    # SSL settings
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    # Security headers
    add_header X-Frame-Options DENY always;
    add_header X-Content-Type-Options nosniff always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # Gzip
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml text/javascript;
    gzip_min_length 1000;

    # Request size (for file uploads if any)
    client_max_body_size 10m;

    # API proxy
    location / {
        limit_req zone=api burst=50 nodelay;

        proxy_pass http://ovni_api;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Connection "";

        # Timeouts
        proxy_connect_timeout 10s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;

        # SSE / streaming support (for /v1/chat)
        proxy_buffering off;
        proxy_cache off;
    }

    # Health check (no rate limit)
    location = /health {
        proxy_pass http://ovni_api;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header Connection "";
    }

    # Static files for dashboard (if built)
    # Uncomment when frontend apps are deployed:
    # location /dashboard/ {
    #     alias /opt/ovni-ai/app/dashboard/dist/;
    #     try_files $uri $uri/ /dashboard/index.html;
    # }
}
```

Deploy the Nginx config to the EC2 instance:
```bash
ssh -i ovni-ai-key.pem ubuntu@<ELASTIC_IP>

# Create directories
sudo mkdir -p /var/www/certbot
sudo mkdir -p /etc/nginx/sites-available
sudo mkdir -p /etc/nginx/sites-enabled

# Copy the config (replace <DOMAIN> with actual domain)
sudo cp /opt/ovni-ai/app/infra/nginx/ovni-ai.conf /etc/nginx/sites-available/ovni-ai

# Replace <DOMAIN> placeholder with actual domain
sudo sed -i 's/<DOMAIN>/api.ovni.ai/g' /etc/nginx/sites-available/ovni-ai

# Enable the site
sudo ln -sf /etc/nginx/sites-available/ovni-ai /etc/nginx/sites-enabled/ovni-ai

# Remove default site
sudo rm -f /etc/nginx/sites-enabled/default

# Test config (will fail on SSL cert paths — that's OK, we'll get the cert next)
# For now, temporarily comment out the HTTPS server block and SSL lines
# OR create a temporary HTTP-only config first:

# Create temporary HTTP-only config for Certbot
sudo tee /etc/nginx/sites-available/ovni-ai-temp << 'TMPEOF'
server {
    listen 80;
    server_name <DOMAIN>;

    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_buffering off;
    }
}
TMPEOF

# Replace <DOMAIN>
sudo sed -i 's/<DOMAIN>/api.ovni.ai/g' /etc/nginx/sites-available/ovni-ai-temp

# Use temp config first
sudo ln -sf /etc/nginx/sites-available/ovni-ai-temp /etc/nginx/sites-enabled/ovni-ai
sudo nginx -t
sudo systemctl reload nginx

# Verify HTTP proxy works

# Should return: {"status":"ok",...}
```

<task id="3.5">
<title>Install SSL certificate with Certbot (Let's Encrypt)</title>
<read_first>
- infra/nginx/ovni-ai.conf (the full HTTPS config to deploy after cert is obtained)
- .planning/phases/01-aws-infrastructure-deploy/01-CONTEXT.md (decision D-10: SSL via ACM or Let's Encrypt)
</read_first>
<action>
Using Let's Encrypt via Certbot is simpler for a single EC2 instance than ACM (which requires a load balancer). Install and obtain the certificate:

```bash
ssh -i ovni-ai-key.pem ubuntu@<ELASTIC_IP>

# Install Certbot
sudo apt install -y certbot python3-certbot-nginx

# Obtain certificate (Nginx must be running with the temp HTTP config)
sudo certbot --nginx -d <DOMAIN> --non-interactive --agree-tos --email ops@ovni.ai --redirect

# Certbot will:
# 1. Verify domain ownership via HTTP challenge
# 2. Obtain the certificate
# 3. Modify the Nginx config to add SSL
# 4. Set up auto-renewal

# ALTERNATIVE: If you prefer manual config (not letting Certbot edit Nginx):
# sudo certbot certonly --webroot -w /var/www/certbot -d <DOMAIN> --non-interactive --agree-tos --email ops@ovni.ai
# Then deploy the full ovni-ai.conf with SSL paths

# After Certbot succeeds, deploy the full Nginx config:
sudo cp /opt/ovni-ai/app/infra/nginx/ovni-ai.conf /etc/nginx/sites-available/ovni-ai
sudo sed -i 's/<DOMAIN>/<ACTUAL_DOMAIN>/g' /etc/nginx/sites-available/ovni-ai
sudo ln -sf /etc/nginx/sites-available/ovni-ai /etc/nginx/sites-enabled/ovni-ai

# Test and reload
sudo nginx -t
sudo systemctl reload nginx

# Verify SSL
curl -s https://<DOMAIN>/health
# Should return: {"status":"ok","timestamp":"..."}

# Verify auto-renewal
sudo certbot renew --dry-run
# Should succeed

# Verify certificate
echo | openssl s_client -connect <DOMAIN>:443 -servername <DOMAIN> 2>/dev/null | openssl x509 -noout -dates
# Should show valid dates
```

Certbot auto-renewal is handled by a systemd timer (`/etc/systemd/system/certbot.timer`) installed automatically.
</action>
<acceptance_criteria>
- Certificate exists: `sudo ls /etc/letsencrypt/live/<DOMAIN>/fullchain.pem` exits 0
- `curl -s https://<DOMAIN>/health` returns `{"status":"ok"`
- No browser SSL warnings when visiting `https://<DOMAIN>/health`
- `sudo certbot renew --dry-run` exits 0 (auto-renewal works)
- HTTP redirects to HTTPS: `curl -sI http://<DOMAIN>/health` returns `301` or `302` with `Location: https://`
</acceptance_criteria>
</task>

# ─── OVNI AI Deploy Script ─────────────────────────────────────

SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"

APP_DIR="/opt/ovni-ai/app"

echo "========================================="

<title>Configure OpenClaw Docker containers to reach metering proxy</title>
<read_first>

<action>
On the EC2 instance, OpenClaw containers need to reach the metering proxy running on the host at port 3001. Unlike Docker Desktop (macOS), Linux Docker uses the host's actual IP.

Verify the setup:
```bash
ssh -i ovni-ai-key.pem ubuntu@<ELASTIC_IP>

# Get the EC2 private IP
PRIVATE_IP=$(hostname -I | awk '{print $1}')
echo "EC2 Private IP: $PRIVATE_IP"

# The METERING_PROXY_URL in .env should be set to reach from Docker containers
# On Linux, containers can use the host gateway:
# Option 1: Use host.docker.internal (Docker 20.10+)
# Option 2: Use the EC2 private IP

# Test with a dummy container
docker run --rm --network ovni-ai-instances alpine wget -qO- "http://host.docker.internal:3001/health" 2>/dev/null || echo "host.docker.internal not available"

# If host.docker.internal doesn't work, use the private IP
docker run --rm --network ovni-ai-instances alpine wget -qO- "http://${PRIVATE_IP}:3001/health" 2>/dev/null || echo "Try adding --add-host"

# For Docker 20.10+, enable host.docker.internal on Linux:
# When launching containers, add: --add-host=host.docker.internal:host-gateway
# This is typically done in instanceOrchestrator.ts
```

Update the `.env` file on the server if needed:
```bash
# If host.docker.internal works:
# METERING_PROXY_URL=http://host.docker.internal:3001

# If using private IP:
# METERING_PROXY_URL=http://<PRIVATE_IP>:3001

# Edit .env
nano /opt/ovni-ai/.env
# Set METERING_PROXY_URL to whichever method works

# Restart app to pick up changes
pm2 restart ovni-ai
```

The `instanceOrchestrator.ts` service passes `ANTHROPIC_API_BASE` (which equals `METERING_PROXY_URL`) to each OpenClaw container as an environment variable. No code changes are needed — only the `.env` value matters.
</action>
